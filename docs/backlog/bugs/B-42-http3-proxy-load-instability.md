# B-42: HTTP/3 逆プロキシ経路が高並行負荷でエラー混入・レイテンシ劣化

## 事象

F-114 の網羅計測で新設した `h3_proxy` 構成（HTTP/3 で受け、HTTP/1.1 バックエンド
`perf-backend` へ中継）を h2load QUIC（`--alpn-list=h3 -n 30000 -c100 -m10`）で計測すると、
`veil:glibc` で **反復間のばらつきが大きく（396/585/651 req/s）非 2xx が混入**
（iteration 1..3 で 17 / 2 / 425 件）、平均レイテンシ 1.48s と劣化した。

対照的に:
- **HTTP/3 File 経路（`h2_1_feat_http3`）は 429.7 req/s・stdev 0.9・エラー 0** で安定。
- `veil:musl` の `h3_proxy` は概ね安定（0〜6 エラー）。

→ HTTP/3 の受信・QUIC 層そのものではなく、**HTTP/3 → バックエンド中継（Proxy ホップ）** 特有の
不安定性である可能性が高い。

## 影響

- HTTP/3 でリバースプロキシ構成を高並行で使うと一部リクエストが失敗し得る（機能影響）。
- 計測上のノイズ源にもなる。

## 調査メモ（未確定・要再現）

- glibc と musl で挙動差があることから、上流 DNS 解決・接続確立タイミング・アロケータ
  （mimalloc vs system）の影響も疑われる。計測ハーネスは上流ホスト名を IP 置換して
  DNS を回避しているため（`start_veil`）、DNS 解決そのものは主因ではない可能性。
- QUIC 高並行（100 接続 × 多重化）× バックエンド接続プール（H2C/HTTP プール）や
  フロー制御・バックプレッシャ（F-32 アクターモデル）の相互作用が疑われる。
- 1.48s の高レイテンシは、失敗リクエストのタイムアウト待ち（`BACKEND_HEADER_TIMEOUT` 等）が
  平均を押し上げている可能性。

## 改修案（候補）

1. `h3_proxy` を低並行（`-c10`）から段階的に上げ、エラー発生の閾値と種別（4xx/5xx/接続断）を特定。
2. サーバログ（`level=debug`）で失敗リクエストの経路（バックエンド接続失敗 / フロー制御 /
   タイムアウト）を切り分け。
3. HTTP/3 → HTTP/1.1 中継のバックプレッシャ・接続プール上限・タイムアウト設定を見直す。
