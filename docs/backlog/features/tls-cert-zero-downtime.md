# バックログ: 0 ダウンタイム TLS 証明書更新

## 目的

稼働中の Veil が **既存接続を不必要に切らず**、または **グレースフルに新証明書へ移行**できるようにする。

## 背景

- 現状は設定リロード（SIGHUP）やプロセス再起動に依存する運用が想定されうる。
- kTLS / rustls の組み合わせでは、セッションと証明書の寿命管理が複雑になりやすい。

## スコープ案

1. **証明書のホットリロード**
   - ディスク上の `cert.pem` / `key.pem`（または設定で指定したパス）変更を検知、または SIGHUP / 専用シグナルで **新規 TLS ハンドシェイクのみ**新証明書を使う。
   - 既存 TCP 接続の扱い: 維持 / 期限後クローズ / ALPN による方針を文書化。

2. **rustls / kTLS の整合**
   - kTLS 有効時は「ハンドシェイクは userspace、データプレーンは kernel」の前提で、証明書ローテ後の挙動を定義。

3. **運用シナリオ**
   - 短寿命証明書（ACME 連携時）との組み合わせを見越し、[acme.md](acme.md) と設計を揃える。

## 受け入れ条件（案）

- ローテーション中に **新規接続が失敗しない**（同一ホスト名・正しいチェーン）。
- README に運用手順（ファイル配置、シグナル、監視）がある。

## リスク

- 秘密鍵の再読み込みタイミングとファイルパーミッション。
- 古いクライアントとの暗号スイート互換。
