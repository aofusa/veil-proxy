# F-49: 設定ファイル・TLS 証明書リロードの正常性確認 E2E テスト

- **優先度**: P1
- **対応状況**: 未着手
- **出典**: ユーザー要求（リロード処理が正しく実装・動作していることの E2E 検証）

## 機能説明・現状

- 設定ホットリロード（SIGHUP / 定期チェック）と TLS 証明書リロード（`src/tls_reload.rs`: mtime 変化検知 + `reload_now()`）は実装済みで単体テストは存在するが、**実プロセスに対する E2E 検証が無い**。
- 「リロード後に新しい設定・新しい証明書で実際にサービングされるか」「リロード中に既存接続が切れないか」「不正な設定・証明書を掴んだ場合に旧状態が維持されるか」が E2E で担保されていない。

## 改修内容

1. E2E テストを追加:
   - 設定リロード: ルーティング / ヘッダ変換等の設定を書き換え → SIGHUP → 新設定が反映されることを検証。
   - 証明書リロード: 証明書ファイルを差し替え → リロード → クライアントが新証明書を観測することを検証。
   - 異常系: 壊れた設定 / 証明書でリロード → 旧状態でサービング継続を検証。
2. E2E で発見したバグは `docs/backlog/bugs/` にチケットを追加し修正する。

## 受け入れ条件

- 上記シナリオの E2E テストが `./tests/e2e_setup.sh test` で通過。
- リロード中のリクエスト断が発生しない（ゼロダウンタイム、F-03 の回帰確認を兼ねる）。

## 依存・リスク

- E2E からプロセスへのシグナル送出・ファイル差し替えの段取りが必要（e2e_setup.sh との連携）。
