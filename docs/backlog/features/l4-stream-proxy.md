# バックログ: L4 (TCP/UDP) ストリームプロキシ

## 目的

nginx stream や Envoy TCP proxy に相当する L4 レイヤーのプロキシ機能を提供し、HTTP 以外のユースケース（DB 接続、gRPC 以外のバイナリプロトコルなど）をサポートしてプロキシとしての用途を拡大する。前回の競合比較で指摘された「L4（raw TCP/UDP）プロキシ機能の完全欠如」という最大の弱点を埋めるための必須機能。

## スコープ案

1. **TCP プロキシ**
   - 既存の LB 実装（RR, LeastConn 等）を再利用した TCP レベルのロードバランシング。
   - 接続数制限（Connection Limiting）。
   - シンプルな TLS termination / passthrough。
2. **TCP ヘルスチェック**
   - TCP 接続の確立可否による upstream 監視。
3. **UDP 拡張（将来）**
   - 初回は TCP を優先し、アーキテクチャ上で UDP のサポートを見据えた設計とする。
4. **観測・運用**
   - Admin/Prometheus での L4 トラフィック統計情報の取得。

## 受け入れ条件（案）

- TCP バックエンドに対する正常なルーティング・ロードバランシングが実行できる。
- E2E テストで、非 HTTP 通信（例: Redis 等）のプロキシが動作することを実証できる。
- TLS passthrough と termination の両方が設定・動作可能。

## 依存・リスク

- L4 プロキシの実装は gRPC 完了後に行うことでアーキテクチャのブレを防ぐ。
- F-04 の dynamic config と連携しやすい設定構造にする。
- HTTPに特化していたルーティング構造の抽象化が必要になるため、既存コードへの影響に注意。
