# バックログ: E2E 試験と実装状況の確認・修正

## 目的

- E2E が **実装の真実** と一致し、回帰の安全網として機能する状態にする。
- 「テストはあるがスキップ・未完成コメントのみ」な箇所を減らす。

## 現状の入口

- [tests/e2e_setup.sh](../../../tests/e2e_setup.sh): 全機能有効 debug ビルド、証明書・プロセス管理。
- [tests/e2e_tests.rs](../../../tests/e2e_tests.rs): シナリオ本体（ファイルが大きいため変更時は差分を明確に）。

## スコープ案

1. **実装状況の棚卸し**
   - E2E 内の `TODO` / `実装には〜が必要` 系コメントを一覧化し、Issue または本バックログに紐づける。
   - SIGHUP リロード、HTTP/3、gRPC、WASM など **feature 境界ごと**にカバレッジマトリクスを作る。

2. **安定性**
   - フレーク対策: 待ち時間の固定 sleep 削減、準備完了の明示待ち、ポート競合の扱い。
   - CI 想定: ヘッドレス Linux、kTLS 非対応環境でのスキップ条件を文書化。

3. **カバレッジ拡大**
   - キャッシュ、圧縮、レート制限、異常系（バックエンドダウン）、メトリクスエンドポイントなど、README の Features と対応表。

## 受け入れ条件（案）

- 「スキップ理由が文書にない `#[ignore]`」をゼロまたは正当化付きにする。
- 最低 1 系の CI コマンド（例: `cargo test` + 条件付き `e2e_setup.sh`）が README / AGENTS に記載と一致。

## リスク

- E2E 実行時間増加 → 並列度・シナリオ分割で緩和。
