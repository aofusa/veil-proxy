# バックログ: サーキットブレーカー・リトライポリシー・異常検知

## 目的

バックエンド障害時に **カスケード故障を防ぎ**、利用者に予測可能なエラー（早い失敗・フォールバック）を返す。

## スコープ案

1. **サーキットブレーカー**
   - upstream 単位で状態機械: Closed / Open / Half-Open。
   - 閾値: 連続 5xx、タイムアウト率、接続失敗など。設定は `config.toml` で表現可能に。

2. **リトライポリシー**
   - べき等メソッドのみ、または明示ヘッダでのリトライ許可。
   - 最大回数、バックオフ（指数 / 固定）、リトライ可能ステータスコードのリスト。

3. **異常検知（アウトライア検出）**
   - スライディングウィンドウでのレイテンシ・エラー率ベースで upstream を一時除外（Envoy の outlier detection 相当のサブセットから開始）。

4. **観測**
   - ブレーカー開閉回数、リトライ回数を Prometheus（将来 OTel）へ。

## 受け入れ条件（案）

- 単体テストで状態遷移が検証できる。
- README に設定例と「リトライが危険なケース」の注意書き。

## 設計上の注意

- リトライは **二重送信** リスク（非べき等 POST）を増やすため、デフォルト保守的に。

## 依存

- ヘルスチェック実装との役割分担を明確化（ヘルスは「ゆっくり」、ブレーカーは「速い失敗」）。
