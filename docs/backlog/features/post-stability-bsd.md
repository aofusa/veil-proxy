# フェーズ 2: BSD 系 OS 対応

## 目的

FreeBSD / OpenBSD 等でも **可能な範囲で Veil を動かす**（Linux 専用機能は段階的に切り替え）。

## 課題（想定）

- **io_uring / monoio**: Linux 依存が強い。BSD では **別ランタイムパス**（従来の epoll/kqueue ベース）が必要になる可能性。
- **kTLS / splice / sendfile / Landlock / 一部 seccomp**: OS ごとに API・セマンティクスが異なる。
- **SO_REUSEPORT + CBPF**: FreeBSD のソケットオプション差異。

## スコープ案

1. **コンパイル方針**
   - `cfg(target_os = "linux")` で既存高速パスを囲み、BSD は **機能縮小モード**（HTTP/1 + rustls のみ等）から開始する案。

2. **CI**
   - QEMU またはネイティブランナーでのスモークテスト（コスト高のためフェーズ 2）。

3. **ドキュメント**
   - 「BSD でサポートされる機能マトリクス」を明示し、期待値を下げるより **誠実に非対応を書く**。

## 受け入れ条件（案）

- 少なくとも 1 系 BSD で `cargo build`（限定 feature）が文書化された手順で成功。
- Linux とのパリティを謳わない（パフォーマンス数字の混同防止）。

## 依存

- aarch64 / コンテナ化より後でもよい大規模タスク。アーキテクチャ分離が進んでから着手すると差分が小さい。
