# F-40: L4 プロキシの splice パイプをスレッドローカルプールで再利用

- **優先度**: P2
- **対応状況**: 完了（2026-07-02）

## 完了メモ

- `src/l4/proxy.rs` にスレッドローカルプール `L4_PIPE_POOL`（上限 64 本 = 同時 32 接続分）と `acquire_pipe()` / `release_pipe()` を実装。`bidirectional_forward` の接続ごと `Pipe::new()`（pipe2(2)）を排除。
- 返却時に `FIONREAD` で残データを検査し、空を確認できた場合のみ再利用（残データあり・ioctl 失敗・満杯時は破棄）。データ混線を構造的に防止。
- 単体テスト 3 件追加（クリーン再利用・汚染破棄・上限遵守）。L4 E2E はまとめ実行で確認。
- **出典**: `docs/artifacts/analysis_results.md` 改善案3

## 機能説明・現状

`src/l4/proxy.rs` の `handle_l4_proxy_connection` は接続ごとに `Pipe::new()`（内部で `pipe2(2)` システムコール）を発行して splice 用パイプを確保している。これはホットパス（接続ごと）でカーネルリソース（fd）確保コストを毎回支払っており、`src/pool.rs` の `get_splice_pipe()`（スレッドローカルにキャッシュしたパイプを再利用）を使う HTTP 側と非対称。

## 改修内容

- L4 のパイプ確保をスレッドローカルプールからの取得・返却に変更し、`pipe2(2)` の毎接続呼び出しを排除する。
- L4 は双方向（up/down）で 2 本のパイプを使うため、プール API が複数本の貸出/返却に対応していることを確認（必要なら拡張）。

## 改修案

- `src/pool.rs` の `get_splice_pipe()`（または L4 向けの同等プール）を利用する。
- 接続終了時にパイプをドレインしてプールへ返却。返却前にパイプ内に残データが無いことを保証する（残ると次接続にリークするため、エラー時は破棄）。

## 受け入れ条件

- L4 ホットパスで接続ごとの `pipe2(2)` が無くなる。
- L4 TCP/TLS パススルーの E2E（双方向転送・大容量）が通過。
- パイプ再利用時にデータ混線が起きないことをテストで確認。

## 依存・リスク

- パイプ返却時の残データ・状態リセット漏れは接続間データ混線の重大バグになる。返却条件を厳格にする。
