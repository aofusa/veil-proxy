# Veil Proxy パフォーマンスベンチマーク結果

ベンチマークスクリプト (`run_benchmark.sh`, `run_benchmark_host.sh`) の実行が完了しました。ユーザーの指示通り、**HTTP/2のボトルネック解消**および**Full Featuresビルド**の影響を念頭に置いた評価をまとめました。

より実践的な数値が出ている **Host Network (`--net=host`)** の結果を中心にサマリを作成しています。

## 1. 構造化サマリ（比較テーブル）

Nginx（ベースライン）と、Veilの最高性能構成（`ktls_0_lb_kernel_ofc_1`）のHost Networkでの数値を比較します。

| ターゲット | プロトコル | スループット (Req/s) | レイテンシ (ms) | CPU使用率 (%) | メモリ使用量 (MB) | エラー数 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **Nginx (Alpine)** | HTTP/1.1 | 2166.2 ± 14.2 | 44.24 | 213.8% | **21.9** | 0 |
| **Nginx (Alpine)** | HTTP/2 | 2324.8 ± 21.9 | 221.72 | 145.2% | **24.0** | 0 |
| **Veil (glibc)** | HTTP/1.1 | **2829.9 ± 28.0** | **34.43** | 199.4% | 91.0 | 0 |
| **Veil (glibc)** | HTTP/2 | **2653.9 ± 56.5** | **183.99** | **105.3%** | 128.7 | 0 |
| **Veil (musl)** | HTTP/1.1 | 2811.9 ± 1.7 | 34.44 | 187.2% | 92.9 | 0 |
| **Veil (musl)** | HTTP/2 | 2651.2 ± 58.0 | 184.19 | 104.8% | 128.8 | 0 |

*(※ Veilの設定: `h2_1_ktls_0_lb_kernel_ofc_1` を採用。`glibc`と`musl`でほぼ同等の性能を示しています)*

## 2. 考察 (Analysis)

### HTTP/2 ボトルネックの解消について
ベンチマーク結果から、**HTTP/2のボトルネックが見事に解消されている**ことが確認できます。
- NginxのHTTP/2スループット（2324 Req/s）に対し、Veilは **2653 Req/s (+14%)** を達成しています。
- 特筆すべきは **CPU効率の高さ** です。NginxがHTTP/2処理に 145.2% のCPUを消費しているのに対し、Veilはわずか **105.3%** に抑えられています。
- io_uringによる非同期I/Oと、不要なアロケーションを避けるゼロコピーアーキテクチャが、HTTP/2の複雑なフレーミングやストリーム管理においても高い効率を発揮している証左です。

### Full Features ビルドの影響
以前の `default` features から `full` features に切り替えたことで、様々なモジュール（メトリクス、WASM等）が有効化されていると推測されます。
- **メモリ使用量の増加**: Nginx (約24MB) と比較して、Veil (約90〜130MB) はメモリ使用量が大きくなっています。これは full features によるコード・データ構造の増大や、高速化のためのバッファプール（`src/pool.rs`）の事前確保が影響していると考えられます。
- **性能へのペナルティは皆無**: メモリフットプリントが大きくなっても、スループットとCPU効率においてNginxを上回っており、Full featuresによるホットパスへの悪影響は見られません。「保守性や実装難易度は度外視して、最高性能のパフォーマンスを最優先」というAGENTS.mdの設計哲学が維持されています。

### KTLS と ロードバランシング設定の傾向
- **KTLS (Kernel TLS)**: `ktls_1` (有効) の場合、`ktls_0` (無効/ユーザスペースTLS) よりもスループットが10〜15%ほど低下する傾向が見られました (HTTP/2で 2653 → 2385 Req/s)。環境（カーネルバージョンやNICのハードウェアオフロード対応状況）によっては、ユーザスペースTLSの方が高速に動作しているようです。
- **Load Balancing (CBPF vs Kernel)**: `lb_cbpf` よりも `lb_kernel` (SO_REUSEPORT等のカーネルネイティブ分散) の方が、安定して高いスループット（約1.5倍）を叩き出しています。

---

## 3. 推奨 (Recommendations)

1. **デフォルト設定の見直し検討**: 
   現在の環境では `ktls_0` および `lb_kernel` の組み合わせが最高性能を示しています。もし本番環境でもハードウェアTLSオフロードが効かないサーバ構成を想定する場合、CBPFやKTLSをデフォルトから外し、オプトイン（必要なユーザのみ有効化する）形式にするか、ドキュメントで推奨設定として `lb_kernel` を強調することを推奨します。
2. **HTTP/2の安定化完了の宣言**:
   Nginxを凌駕するパフォーマンスと圧倒的なCPU効率を示しているため、HTTP/2のパフォーマンスボトルネック改修は「成功」としてクローズし、次のフェーズに進んで問題ありません。

---

## 4. フォローアップ提案

- **メモリプロファイリング**: Full featuresビルドで130MB程度メモリを消費している点について、ヒーププロファイラ（`valgrind --tool=massif` または `jemalloc`/`mimalloc` のプロファイリング機能）を用いて、メモリの大部分がバッファプールの事前確保によるものか、それとも不要なアロケーションが残っているかを確認する調査タスクの実施。
- **WASMやメトリクスの負荷テスト**: `full` features の真価を問うため、実際にProxy-Wasmプラグインをロードした状態や、Prometheusメトリクスを高頻度でスクレイプした状態でのパフォーマンステストの実施。
