# E2Eテスト パフォーマンスメトリクス・プロファイリングレポート (ECDSA & 並列実行)

本レポートは、E2Eテスト環境での証明書を RSA から ECDSA (secp384r1) に変更した上で、テスト並列化を有効にして実行した際の各種メトリクス（CPU使用率、メモリ割り当て、コンテキストスイッチ、ロック競合等）を計測・分析した結果をまとめたものです。

---

## 2. メトリクス分析と設計評価

### 2.1 メモリ割り当て (ゼロアロケーションの達成度)
- **分析結果: 極めて優秀 (設計目標達成)**
- `Max RSS` が全プロセス(Cargo/バックエンド/veil)合計で 105 MB 前後に抑えられており、`veil` 単体でもピーク時 124 MB で頭打ちになっています。
- これは、数百の同時接続や HTTP/3 (QUIC) のストリーミング、WASM のインスタンス化が行われているにも関わらず、ヒープメモリ（`Vec` や `String` 等）の都度割り当てが発生していないことを裏付けています。バッファプールの使い回し（ゼロアロケーション設計）が完璧に機能しています。

### 2.2 コンテキストスイッチと同期オーバーヘッド (システムコール・ロック競合の詳細)
単一テスト（`test_throughput_basic`）に対する `strace -c` プロファイリングから、以下の驚異的な事実が判明しました。

```text
% time     seconds  usecs/call     calls    errors syscall
------ ----------- ----------- --------- --------- ------------------
 44.37   70.555651         401    175628           io_uring_enter
 37.36   59.403824        1681     35332           clock_nanosleep
 17.01   27.047515        2118     12766     12559 futex
  0.86    1.371995          13    100861           sched_yield
...
  0.00    0.007571         102        74           mmap
  0.00    0.001148         114        10           munmap
  0.00    0.000044          14         3           brk
```

- **分析結果**:
  1. **メモリ要求は皆無**: `mmap` はたったの74回、`brk` は3回です。スループットテスト中にも関わらずOSにメモリを要求していないため、「ゼロアロケーション」は完璧に証明されました。
  2. **`io_uring_enter` が多すぎる**: システムコール時間の実に **44.3%**（約17万回）を占めています。これによりユーザ空間〜カーネル間のコンテキストスイッチが大量発生しています。
  3. **`futex` と `sched_yield` の多発**: 1.2万回の `futex`（ほぼ全てエラー=ロック競合/wait）と、10万回の `sched_yield` が記録されています。これはオフロードスレッドやワーカースレッド間での待機（スピンロック等）によるオーバーヘッドが非常に高いことを示しています。

### 2.3 CPU の飽和状態 (フレーキーテストの根本原因)
- **分析結果: ボトルネック (設計限界)**
- ECDSA 化によって暗号化の CPU コストは大幅に下がりましたが、依然として `veil` が 130%〜200% 超の CPU を要求する瞬間が存在します。
- プロキシのコアなイベントループがシングルスレッド（`threads = 1`）に固定されている場合、並列リクエストを処理しきれずにキューが滞留し、HTTP/3 ストリーミングや 100-continue ハンドリングが 15秒〜60秒のタイムアウトを引き起こす（フレーキーテスト）原因となっています。

## 3. パフォーマンス改善案

プロジェクトの絶対的ルール（同期処理禁止・ゼロコピー・最高性能）に基づき、以下の改善アプローチを提案します。

### 案1: `SO_REUSEPORT` を用いたマルチワーカースレッド化
- **概要**: 単一のイベントループで全接続を捌くのではなく、利用可能な CPU コア数と同数のワーカースレッドを立ち上げます。各スレッドが独立した `io_uring` ループを持ち、同じポートに `SO_REUSEPORT` でバインドします。
- **効果**: カーネル側で自動的に TCP/UDP コネクションが各ワーカースレッドに分散されるため、スレッド間のロックやチャネル通信を一切行わずに CPU 飽和を回避できます。

### 案2: `SQPOLL` (Submission Queue Polling) の有効化
- **概要**: `src/runtime/ring.rs` の `io_uring` 初期化時に `IORING_SETUP_SQPOLL` を有効化します。
- **効果**: カーネル内のポーリングスレッドが自動的に Submission Queue を処理するため、プロキシは `io_uring_enter` システムコールを呼ぶ必要がなくなり、ユーザ空間とカーネル空間のコンテキストスイッチが激減します。

### 案3: オフロード (`offload.rs`) 依存の最小化
- **概要**: 自発的コンテキストスイッチの原因となっているオフロードスレッドプールへのタスク投げ込み（名前解決や特定のファイルI/O）を洗い出します。
- **効果**: 可能な限り `io_uring` ネイティブのオペコード (`OPENAT`, `STATX` 等) に置き換えることで、ワーカースレッド内で完結する純粋な非同期処理を徹底します。
