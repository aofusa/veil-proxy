# Veil-Proxy パフォーマンス測定レポート

本レポートでは、`veil-proxy` (glibc版 / musl版) および比較基準となる `nginx` コンテナを用いたパフォーマンス測定結果を報告します。`veil-proxy` の設定ファイル (`config.toml`) において、主要な機能 (`ktls`, `http2_enabled`, `mimalloc`) のオン・オフを組み合わせた条件でのスループットへの影響を詳細に分析しました。

## 測定環境・条件

- **ツール**: `williamyeh/wrk` コンテナ (HTTP/1.1 ベンチマーク)
- **対象ファイル**: 約650KB の静的ファイル (`assets/www/index.html`)
- **負荷条件**: 2スレッド、10同時接続 (`-t2 -c10 -d5s`)
  - ※ 100同時接続時のテストでは、`veil-proxy` のリバースプロキシ時に大量の502エラーが発生し、正しく測定できない事象が確認されたため、安定して通信を処理できる10同時接続で統一して測定しています。
- **ネットワーク**: `--network host` モード
- **セキュリティ設定**: `seccomp.json` を適用したセキュア環境。ファイルシステムは `read-only` およびセキュアな `--tmpfs` 設定。

## 測定結果 (サマリー)

以下に各コンテナ・構成でのスループット (Requests/sec および Transfer/sec) を示します。

| コンテナ (ビルド) | 設定条件 | Req/Sec | Transfer/Sec | エラー発生状況 |
| :--- | :--- | :--- | :--- | :--- |
| **Nginx (Standalone)** | Baseline | 152.03 | 96.73 MB/s | なし |
| **Veil (glibc)** | Base (全機能ON) | 98.16 | 62.76 MB/s | **あり** (Socket read error: 691) |
| **Veil (glibc)** | **kTLS OFF** | **168.14** | **107.40 MB/s** | なし |
| **Veil (glibc)** | HTTP/2 OFF | 93.13 | 59.32 MB/s | **あり** (Socket read error: 640) |
| **Veil (glibc)** | mimalloc OFF | 87.84 | 55.78 MB/s | **あり** (Socket read error: 748) |
| **Veil (musl)** | Base (全機能ON) | 84.37 | 53.55 MB/s | **あり** (Socket read error: 734) |
| **Veil (musl)** | **kTLS OFF** | **165.58** | **105.65 MB/s** | なし |

※ `Base (全機能ON)` の設定: `ktls = true`, `http2_enabled = true`, (コンパイル時 `mimalloc` 適用)

## パフォーマンス考察・詳細分析

### 1. kTLS (Kernel TLS) の影響（最重要発見）
本測定における最も特筆すべき発見は、**kTLSを無効化（ユーザー空間でのTLS処理）した状態の Veil-Proxy が、Nginx Standalone を超える最高パフォーマンス (約168 Req/sec, 107 MB/s) を叩き出した**点です。
逆に、kTLS が有効な環境ではパフォーマンスが約 40% 低下 (98 Req/sec) し、`wrk` 側で大量の `Socket read error` が発生しました。
- **原因の推測**: 現在のLinuxカーネルや `io_uring` 実装環境において、kTLSによるソケット操作（カーネル内暗号化/復号化）とイベントループの相性に何らかのボトルネック、あるいはTLSレコード境界における早すぎる切断（EOFの誤検知）などが発生していると考えられます。
- ユーザー空間の実装である `rustls` (非kTLS) の方が極めて高速に動作しており、Veil-Proxyが元来目標としている「ゼロコピーかつ高スループット」な設計の優位性が証明されています。

### 2. glibc vs musl の比較
- kTLS無効時の最高性能においては、`glibc` (168.14 Req/s) が `musl` (165.58 Req/s) を僅かに上回りました。
- kTLS有効時（Base）の比較でも、`glibc` (98.16 Req/s) は `musl` (84.37 Req/s) に比べて約16% 高速です。
- 概ね `glibc` の方がメモリアロケーションや各種libcコールの最適化が進んでいるため、高負荷なプロキシ/TLS処理において有利であることが確認できます。

### 3. mimalloc の影響
- `glibc` 環境において、`mimalloc` アロケータを用いた場合のパフォーマンスは、使用しない場合 (87.84 Req/s) よりも高速 (98.16 Req/s) となりました (約11% の向上)。
- これは、Rustのデフォルトアロケータよりも `mimalloc` のヒープ管理が高効率であり、メモリフラグメンテーションの抑制や確保・解放のオーバーヘッドが減少しているためと考えられます。

### 4. 高負荷 (HTTP/1.1) における課題
今回は10同時接続にて安定した結果を得ましたが、事前に100同時接続による測定を試みた際、`veil-proxy` は直ちに大量の `502 Bad Gateway` を返却しました。
原因としては、Nginxバックエンドへの新規プロキシコネクションの確立において、同時多発的なリクエストに対してポート枯渇、もしくは `io_uring` での非同期ソケット確立・破棄が追いついていない可能性（コネクションプーリングの未実装やファイルディスクリプタ上限への到達等）が疑われます。

## 結論
`Veil-Proxy` は、**kTLSをオフにした状態で運用することにより、同等の条件のNginxを凌駕する強力なパフォーマンスを発揮**します。現状では、安定性とスループットの観点から kTLS よりもユーザー空間のTLS (rustls) を採用し、`glibc` と `mimalloc` を組み合わせたバイナリが最適解と言えます。今後は、高並行アクセス（HTTP/1.1 Proxy時）での502エラー改善や、kTLS環境下のソケットエラー原因特定がさらなる性能向上に向けた鍵となります。
