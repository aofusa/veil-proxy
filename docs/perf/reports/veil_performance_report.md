# Veil Proxy パフォーマンス測定レポート

## 1. 概要・検証環境

Veil Proxy のパフォーマンスを、glibc版コンテナおよびmusl版コンテナ間で計測・比較しました。また、比較対象として nginx コンテナの計測も実施しています。

### 計測条件・ルール (§3 証拠基準に準拠)
- **環境**: Dockerコンテナ間でのネットワーク通信 (専用ブリッジネットワーク `perf_net_full` を作成)
- **負荷生成ツール**: 
  - HTTP/1.1: `williamyeh/wrk` (4スレッド、100コネクション、10秒)
  - HTTP/2: `h2load` (30000リクエスト、100コネクション、10マルチプレックス)
- **対象ファイル**: 静的ファイル配信 (`docker/assets/www/index.html`)
- **セキュリティ要件**: seccomp および Landlock は有効化したまま実施。
- **実行方針**: 各条件につき **最低3回の計測を反復** し、中央値 (Median) と標準偏差 (Stdev) を算出しました。

## 2. 探索的設定の組み合わせ

Veil の機能 (`ktls`, `http2`, `mimalloc` 組み込み済) の中から、パフォーマンスに影響を与える可能性のある以下の4つの設定軸を列挙し、網羅的に組み合わせ (2^4 = 16通り) て検証しました。

1. **HTTP/2**: 有効(`h2_1`) / 無効(`h2_0`)
2. **kTLS**: 有効(`ktls_1`) / 無効(`ktls_0`) 
3. **SO_REUSEPORT 分散方式**: CBPF(`lb_cbpf`) / カーネルデフォルト(`lb_kernel`)
4. **OpenFileCache (OFC)**: 有効(`ofc_1`) / 無効(`ofc_0`)

## 4. 考察・分析

### 4.1 glibc版 vs musl版
全体を通して **musl版の方がglibc版に比べて、一貫して高いスループット（Req/s）** を記録しました。
- 最高性能（`h2_1_ktls_0_lb_kernel_ofc_1`）において、glibc版が `2432 Req/s` (HTTP/1.1) に対して musl版が `2688 Req/s` となり、**musl版が約10.5%高速**でした。
- メモリ使用量については、glibcもmuslも大きな差はなく（同構成時で30MB〜50MB前後）、どちらも軽量です。
- 静的コンパイルされたmusl版+mimallocのアロケーションが、コンテナ環境特有のI/O処理やsyscall実行と相性が良い可能性が考えられます。

### 4.2 Nginx との比較
- Nginx（HTTP/1.1：`1665 Req/s`、HTTP/2：`1862 Req/s`）と比較した場合、**最適化された Veil Proxy（musl版の最高設定）は、Nginxの約1.6倍の性能**を記録しています。
- OFC（OpenFileCache）を無効化した通常状態（`lb_kernel_ofc_0` 系）でも、Veil は 1950〜1970 Req/s ほど出ているため、**デフォルト設定でもNginxを超える、あるいは同等の高いパフォーマンスが発揮**されています。

### 4.3 kTLS の影響 (`ktls_0` vs `ktls_1`)
- 今回の計測環境（Dockerコンテナ、vethインターフェース利用）では、**kTLSを有効化するとパフォーマンスが顕著に低下**する結果となりました。
- 例えば musl版・HTTP/1.1・OFC有効時で、kTLS無効なら `2688 Req/s` のところが、kTLS有効時は `1801 Req/s` まで低下します。
- HTTP/2 に至っては `166〜327 Req/s` まで劇的に低下しています。コンテナ上のネットワーク（ソフトウェアブリッジ・veth）ではハードウェアへのTLSオフロードの恩恵が一切受けられないだけでなく、むしろカーネル/ユーザースペース間のデータ移行のオーバヘッドによる著しいボトルネックを生んでいることがわかります。

### 4.4 Load Balancing (CBPF vs Kernel)
- `reuseport_balancing = "cbpf"` は、`"kernel"` と比較して **パフォーマンスの低下（スループットの減少）** をもたらしました。
- 例えば、`h2_0_ktls_0_lb_cbpf_ofc_0` は `1216 Req/s` (musl版) ですが、`lb_kernel` は `1959 Req/s` となっています。
- クライアントIPに基づくCBPF分散は、限られたコネクション（今回は100接続からの集中）からのベンチマークという状況下では、特定のワーカースレッドに処理が偏ってしまうため、分散効率が極端に悪化したものと考えられます。実際の多様なIPからの実運用トラフィックであればキャッシュ効率が上がる可能性はありますが、ベンチマークにおいてはカーネル依存の均等割り当て（`kernel`）が適しています。

### 4.5 OpenFileCache の影響 (`ofc_0` vs `ofc_1`)
- 予想通り、静的ファイルの配信において **OpenFileCache の有効化はスループット向上に寄与** しています。
- CBPF環境での musl版（HTTP/1.1）においては、`1216 Req/s` -> `1643 Req/s` （約35%向上）。
- Kernel分散環境での musl版（HTTP/1.1）においては、`2267 Req/s` -> `2688 Req/s` （約18%向上）。
- OSレイヤーでのメタデータ取得（stat 等）のオーバヘッド削減が如実に効果を表しています。

### 4.6 Host Networking (`--net=host`) の影響
Dockerのデフォルトであるブリッジネットワークとvethペアのオーバヘッドを回避するため、追加で `--net=host`（ホストネットワーク、非特権ポート8443使用）を指定して同条件で測定を行いました。結果は `results_host_summary.md` に保存されています。

- **全体的なスループット向上**: 
  vethをバイパスしたことで、Nginx (HTTP/1.1で約 `1768 Req/s`)、Veil (musl版 HTTP/1.1で最高 `2839 Req/s`) と共にブリッジネットワーク時よりもベースの性能が底上げされました。
- **kTLS の大幅な改善**: 
  ブリッジネットワークでは劇的な速度低下を引き起こしていた kTLS (`ktls = true`) ですが、`--net=host` 環境下では、HTTP/2で `1942 Req/s` (musl版) を記録し、ブリッジ環境（327 Req/s）から大幅に改善しました。ホストのNICとカーネルTCPスタックを直接利用できる環境では、kTLSが本来の性能を発揮しやすくなることが分かります。
- **ユーザー空間TLSとの逆転には至らず**: 
  改善したとはいえ、今回のベンチマーク環境では依然として kTLS無効（Rustlsによるユーザー空間暗号化）の方がHTTP/2で `2183 Req/s` (musl版) と、やや高い性能を示しました。

## 5. 推奨設定と結論

今回のベンチマーク検証からの **コンテナデプロイにおける推奨設定** は以下の通りです。

- **ビルドイメージ**: **musl 版** (`veil:musl`) の採用を推奨します。
- **ネットワークモード**: 可能であれば、パケット処理のオーバヘッドを最小化するために `--net=host` の使用を推奨します。
- **kTLS**: コンテナ内で運用する場合、ブリッジネットワークでは **無効 (`ktls_enabled = false`)** が必須です。ホストネットワークであっても、現状のワークロードでは無効化（ユーザー空間TLS）の方が若干高速です。
- **ロードバランシング**: クライアントIPが均等に分散しない環境（ベンチマーク時や、前段にL4LBがありSNATされている場合など）では **`reuseport_balancing = "kernel"`** を推奨します。
- **キャッシュ**: 静的ファイル配信を利用するパスでは、必ず **`[route.open_file_cache]`** を有効にしてください。

**結論として、ホストネットワークを利用し最適化された構成（`h2_1_ktls_0_lb_kernel_ofc_1`）を使用することで、HTTP/1.1で約2800 Req/s、HTTP/2で約2200 Req/sという、同等のNginxを大きく上回る高スループット（HTTP/1.1で約1.6倍）を実現可能です。**
