# v0.6.0 プラットフォーム×arch 検証マトリクス（ビルド / テスト / E2E / perf）

本ドキュメントは v0.6.0 の「全プラットフォーム×arch のビルド・テスト・E2E・perf」検証の
実施結果と、当開発ホストで実行不能な項目の代替検証・環境制約を記録する。

計測ホスト: x86_64 Linux（4 コア・co-tenant 有り・長期稼働）。KVM は同一アーキ（x86_64）
のみ利用可能で、クロスアーキ（aarch64 等）の full-system VM は TCG のみ。

## 1. ビルド（現行 main コード、cargo warning 0）

| プラットフォーム | x86_64 | aarch64 | ビルド経路 / 備考 |
|---|---|---|---|
| Linux glibc（io_uring 既定） | ✅ | ✅ | x86_64=native、aarch64=cargo-zigbuild（**aws-lc-sys が cmake ビルダー経路のため cmake 導入必須**） |
| Linux glibc（epoll reactor） | ✅ | ✅ | `--features epoll`。バックエンド選択は feature/cfg |
| Linux musl | ✅ | ✅ | rust-musl-cross（aarch64 は aws-lc-sys bindgen に `BINDGEN_EXTRA_CLANG_ARGS` sysroot 指定） |
| macOS（darwin, ring provider） | ✅ | ✅ | cargo-zigbuild。universal2 も生成可 |
| Windows（msvc） | ✅ (ring) | ✅ (aws_lc_rs) | x86_64=ring（cargo-xwin）。**aarch64=aws_lc_rs + cmake**（ARM asm・NASM 不要。ring 0.17 は aarch64-windows で prebuilt asm 無く cargo-xwin の `/imsvc` handling で失敗するため provider を arch 分割） |
| FreeBSD 14.3（aws_lc_rs provider, kTLS/AIO） | ✅ | ✅ | いずれも **VM ネイティブビルド**（`cargo build --release`。cross は aws-lc-sys/std 制約）。**aarch64 は Linux aarch64 と違い FreeBSD arm64 VM-IMAGE が TCG で実用起動する**ため、`pkg install rust cmake` → in-VM ネイティブビルドで検証（§3・§5、`tools/qemu/fbsd-arm64-vm.sh`） |
| OpenBSD（ring provider, pledge/unveil） | ✅ | ⚠️ | x86_64=VM ネイティブビルド（Tier3 = prebuilt std 無し、cross-zigbuild 不可）。aarch64 は ready な VM-IMAGE が無く miniroot からの**対話 install が必要**+Tier3 in-VM ビルドで TCG では非現実的（§5） |

- x86_64: **6 プラットフォームすべてビルド成功**。
- aarch64: Linux(glibc/musl)・macOS・Windows(aws_lc_rs)・**FreeBSD(in-VM ネイティブ)** ビルド成功。OpenBSD aarch64 のみ環境制約（ready イメージ無し）。
- 全 feature マトリクス（24 構成）・clippy（default/full）・fmt は x86_64 Linux で warning 0。

## 3. E2E（プラットフォーム別）

- **Linux io_uring / epoll**: 上記 §2 の E2E スイート（バックエンド実起動、HTTP/1.1・HTTP/2・HTTP/3・gRPC・WebSocket・L4 等 533 ケース）で検証。
- **FreeBSD capability mode 静的配信（F-123）**: VM で index 200 / nested 200 / パストラバーサル 403 / panic 0。
- **FreeBSD kTLS（F-126）**: VM で kTLS on/off 双方の静的配信を検証済み。
- **aarch64 full-system io_uring ライブ E2E**: KVM 不可ホストのため TCG が実用不能（Ubuntu/Alpine とも初期化段階で停滞）。→ aarch64-musl クロスビルド成功 + io_uring アーキ非依存（カーネル ABI 共通）で代替（`tools/qemu/README.md`）。

## 4. perf（tools/perf、docs/perf）

- **Linux x86_64 io_uring 非退行**: `docs/perf/README.md` の「v0.6.0 io_uring 非劣化確認」に

  （`h2_1_ktls_0_lb_kernel_ofc_1`、3 反復、全 Non-2xx=0）を再計測し退行なしを確認:

  | veil_glibc | HTTP/2 | 2848.8 ± 60.1 | 1.19×（nginx 2393.1） |
  | veil_musl | HTTP/1.1 | 3131.9 ± 38.6 | 1.43× |
  | veil_musl | HTTP/2 | 2740.6 ± 67.0 | 1.15× |

  本セッションの変更は `cfg`/FreeBSD 限定で Linux io_uring ホットパス無変更のため、
  値は v0.6.0 公表値と統計的に同一（退行なし）。
- **HTTP/3 / gRPC / WebSocket・kTLS ホスト計測**: `docs/perf/README.md` に既存記録

## 5. aarch64 full-system の実機検証状況（当ホスト = クロスアーチ TCG のみ）

当ホストは KVM がクロスアーチに使えず aarch64 は TCG エミュレーションのみ。OS により
TCG 適性が大きく異なる:

- **FreeBSD 14.3 arm64: TCG で実用起動 ✅**（Linux aarch64 と対照的）。VM-IMAGE を
  qemu-system-aarch64（`-cpu cortex-a72`）で起動し multi-user・network・sshd 到達を確認。
  aarch64-unknown-freebsd は Rust Tier3（cross は build-std 要）のため **VM 内で
  `pkg install rust cmake` → `cargo build --release` ネイティブビルド**し、HTTPS 静的
  配信を smoke 実行して aarch64 の「ビルド + 動作確認」を達成（`tools/qemu/fbsd-arm64-vm.sh`、
  provision は single-user で SSH 鍵注入 / growfs は single-user ro で fsck→growfs）。
  → これが **aarch64 プラットフォームの E2E（動作確認）を代表**する。
- **Linux aarch64: TCG で実用不能 ❌**（Ubuntu/Alpine とも初期化段階で氷河化/ソフトロック。
  `tools/qemu/README.md`）。ビルドは aarch64-musl/gnu クロスで検証、io_uring はアーキ非依存。
- **OpenBSD aarch64: 未実施**（ready な VM-IMAGE が無く miniroot からの対話 install が必要、
  かつ Tier3 in-VM ビルドで TCG では非現実的。x86_64 OpenBSD は VM 検証済み）。
- **macOS / Windows（両 arch）**: 対応ホストが無くネイティブ実行・E2E・perf は不可
  （クロスビルド + packaging のみ。ネイティブセキュリティ機構は保守的プロファイル）。
- **フル E2E スイート（tests/e2e_tests.rs 533 ケース）の aarch64 実行**: クロスコンパイルした
  テストバイナリ + TCG 実行はコスト過大のため、aarch64 は smoke E2E で代表する。
